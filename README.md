# JWT Auth Demo — спрощена версія

Це невеликий навчальний проєкт на FastAPI, який показує, як працює JWT-автентифікація.
У ньому реалізовані реєстрація користувачів, логін, оновлення токенів через refresh-token, вихід із системи, ролі користувачів та видалення акаунта.
Для зберігання даних використовується звичайний словник у пам'яті програми, тому справжня база даних тут не потрібна

---

## Структура проєкту

```
main.py          — основний файл застосунку
tests/
  test_main.py   — тести для перевірки роботи
requirements.txt
README.md
```

## Встановлення та запуск

Створити та активувати віртуальне середовище:

```bash
python3 -m venv venv
source venv/bin/activate
# Windows:
venv\Scripts\activate
```

Встановити залежності:

```bash
pip install -r requirements.txt
```

Запустити сервер:

```bash
uvicorn main:app --reload
```

Після запуску застосунок доступний за адресою: `http://127.0.0.1:8000`  
Swagger-документація: `http://127.0.0.1:8000/docs`

Під час запуску автоматично створюється адміністратор:
- **username:** admin
- **password:** admin123
- **role:** admin

## Основні ендпоінти

Базовий URL: `http://127.0.0.1:8000`

---

### POST /register

Реєстрація нового користувача. Потрібно передати ім'я, email і пароль. У разі успіху повертаються дані нового користувача. Якщо такий username уже існує — помилка 400.

**Запит:**
```json
{
  "username": "ivan",
  "email": "ivan@example.com",
  "password": "securepass123"
}
```

**Відповідь `201 Created`:**
```json
{
  "username": "ivan",
  "email": "ivan@example.com",
  "role": "user"
}
```

**Помилка `400 Bad Request`** (username уже зайнятий):
```json
{
  "detail": "Користувач з таким username вже існує"
}
```

---

### POST /login

Вхід у систему. Надсилається як `application/x-www-form-urlencoded` (поля `username`, `password`). Повертає `access_token` (15 хвилин) та `refresh_token` (7 днів).

**Відповідь `200 OK`:**
```json
{
  "access_token": "eyJhbGciOiJIUzI1NiIsInR5cCI6IkpXVCJ9...",
  "refresh_token": "eyJhbGciOiJIUzI1NiIsInR5cCI6IkpXVCJ9...",
  "token_type": "bearer"
}
```

**Помилка `401 Unauthorized`:**
```json
{
  "detail": "Невірний username або пароль"
}
```

---

### POST /refresh

Отримати нову пару токенів за допомогою refresh-токена. Якщо токен прострочений або недійсний — помилка 401.

**Запит:**
```json
{
  "refresh_token": "eyJhbGciOiJIUzI1NiIsInR5cCI6IkpXVCJ9..."
}
```

**Відповідь `200 OK`:**
```json
{
  "access_token": "eyJhbGciOiJIUzI1NiIsInR5cCI6IkpXVCJ9...",
  "refresh_token": "eyJhbGciOiJIUzI1NiIsInR5cCI6IkpXVCJ9...",
  "token_type": "bearer"
}
```

**Помилка `401 Unauthorized`** (передано access-токен замість refresh):
```json
{
  "detail": "Очікувався refresh-токен"
}
```

---

### POST /logout

Вихід із системи. Заголовок: `Authorization: Bearer <access_token>`. Тіло запиту не потрібне. Поточний access-токен додається до чорного списку.

**Відповідь `200 OK`:**
```json
{
  "message": "Вихід виконано успішно"
}
```

**Помилка `401 Unauthorized`** (повторне використання відкликаного токена):
```json
{
  "detail": "Токен анульовано (logout)"
}
```

---

### GET /profile

Повертає інформацію про поточного користувача. Заголовок: `Authorization: Bearer <access_token>`.

**Відповідь `200 OK`:**
```json
{
  "username": "ivan",
  "email": "ivan@example.com",
  "role": "user"
}
```

**Помилка `401 Unauthorized`:**
```json
{
  "detail": "Не вдалося перевірити токен"
}
```

### POST /change-password

Зміна пароля. Заголовок: `Authorization: Bearer <access_token>`.

**Запит:**
```json
{
  "old_password": "securepass123",
  "new_password": "newsecurepass456"
}
```

**Відповідь `200 OK`:**
```json
{
  "message": "Пароль успішно змінено"
}
```

**Помилка `401 Unauthorized`:**
```json
{
  "detail": "Поточний пароль вказано невірно"
}
```

---

### DELETE /profile

Видалення акаунта. Заголовок: `Authorization: Bearer <access_token>`. Для підтвердження потрібно вказати пароль.

**Запит:**
```json
{
  "password": "securepass123"
}
```

**Відповідь `200 OK`:**
```json
{
  "message": "Акаунт видалено"
}
```

**Помилка `401 Unauthorized`:**
```json
{
  "detail": "Пароль вказано невірно — акаунт не видалено"
}
```

---

### GET /admin/users

Доступний тільки для користувачів із роллю `admin`. Повертає список усіх зареєстрованих користувачів. Заголовок: `Authorization: Bearer <access_token>`.

**Відповідь `200 OK`:**
```json
[
  {
    "username": "admin",
    "email": "admin@example.com",
    "role": "admin"
  },
  {
    "username": "ivan",
    "email": "ivan@example.com",
    "role": "user"
  }
]
```

**Помилка `403 Forbidden`** (звичайний користувач):
```json
{
  "detail": "Недостатньо прав доступу"
}
```

### Помилка валідації (загальний приклад)

**Помилка `422 Unprocessable Entity`** (наприклад, пароль коротший за 8 символів):
```json
{
  "detail": [
    {
      "type": "string_too_short",
      "loc": ["body", "password"],
      "msg": "String should have at least 8 characters",
      "input": "123"
    }
  ]
}
```

## Коди помилок

| Код | Опис |
|-----|------|
| 400 | Користувач уже існує |
| 401 | Неправильний логін або пароль, невалідний чи прострочений токен |
| 403 | Недостатньо прав доступу |
| 422 | Помилка перевірки вхідних даних |

## Теоретична частина

### Що таке JWT і з яких частин складається токен

JWT (JSON Web Token) — компактний формат токена для передачі даних між клієнтом і сервером, що найчастіше використовується для автентифікації та авторизації. Токен складається з трьох частин, розділених крапками:

**Header** — тип токена (JWT) та алгоритм підпису, наприклад HS256.

**Payload** — основні дані: ідентифікатор користувача, час завершення дії, додаткові поля. Ці дані можна прочитати, тому конфіденційну інформацію тут зберігати не слід.

**Signature** — цифровий підпис на основі Header і Payload із використанням секретного ключа. Захищає дані від стороннього редагування.

---

### Що відбувається у `create_access_token()`

Функція копіює передані дані, щоб не змінювати оригінальний словник. Визначає час дії токена і додає поля `exp` та `type: access`. Після цього підписує все через `jwt.encode()` з `SECRET_KEY` та алгоритмом HS256 і повертає готовий JWT-токен.

---

### Навіщо потрібен `SECRET_KEY`

`SECRET_KEY` використовується для підпису та перевірки JWT-токенів — сервер переконується, що токен створено саме ним і він не був змінений. Якщо ключ потрапить до зловмисника, той зможе самостійно створювати валідні токени для будь-яких користувачів, включно з адміністратором.

Тому `SECRET_KEY` потрібно зберігати в змінних середовища або спеціальних сховищах секретів, а не в коді. Ключ також повинен бути достатньо довгим і випадковим.

---

### Чому пароль зберігається у хешованому вигляді: bcrypt vs SHA-256

Паролі не можна зберігати у відкритому вигляді: у разі витоку бази даних вони одразу стануть доступні стороннім. Хешування зберігає лише результат математичного перетворення.

**bcrypt** вважається кращим для паролів з таких причин:

- Автоматично використовує сіль — однакові паролі матимуть різні хеші.
- Працює навмисно повільно, що значно ускладнює brute force.
- Розроблений саме для роботи з паролями.

**SHA-256** навпаки створений для швидкої роботи і є універсальною хеш-функцією — тому його легше використовувати для масового перебору паролів.

---

### Dependency Injection у FastAPI на прикладі `get_current_user()`

FastAPI підтримує механізм Dependency Injection: якщо в параметрах маршруту вказано `current_user = Depends(get_current_user)`, то перед виконанням маршруту FastAPI спочатку викликає `get_current_user()`.

Ця функція отримує токен із заголовка `Authorization`, перевіряє його, знаходить відповідного користувача та повертає його дані. FastAPI автоматично передає результат у параметр `current_user`. Логіку перевірки токена не потрібно дублювати в кожному маршруті — код стає чистішим.

### Вразливості безпеки

Навіть у невеликому навчальному проєкті є кілька потенційних проблем:

- **Секретний ключ у коді** — якщо репозиторій стане публічним, будь-хто зможе створювати власні токени.
- **Відсутнє обмеження спроб входу** — можливий brute force.
- **Дані лише в пам'яті** — після перезапуску сервера всі користувачі та чорний список токенів зникають.
- **Refresh-токени не відкликаються автоматично** — у разі крадіжки ними можна користуватися до закінчення терміну дії.
- **Немає обов'язкового HTTPS** — токени можуть бути перехоплені під час передачі.
- **Слабка перевірка пароля** — контролюється лише мінімальна довжина без вимог до складності.

Цей проєкт добре підходить для навчання принципам JWT, але для продакшн-використання потрібно додати: захищене зберігання секретів, повноцінну базу даних, HTTPS, обмеження спроб входу та інші механізми безпеки.
